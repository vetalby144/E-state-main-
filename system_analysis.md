# Аналіз системи та визначення потенційних місць відмови

## 1. Опис архітектури системи
Система E-state — це веб-застосунок для перегляду об’єктів нерухомості, створення заявок та керування оголошеннями.  
Архітектура включає:

- Flask — основний веб-фреймворк.
- Gunicorn — WSGI-сервер у production.
- SQLite — база даних у тестовому середовищі (ephemeral storage Render).
- HTML шаблони (Jinja2).
- Моделі User / Property / Inquiry.
- Seed-скрипт для ініціалізації тестових даних.
- Render.com — інтернет-хостинг для PRODUCTION та DEVELOP середовищ.

---

## 2. Потенційні точки відмови системи

### 2.1. База даних
**Проблеми:**
- SQLite зберігається у тимчасовій файловій системі (ephemeral FS) на Render.
- Дані стираються при кожному перезапуску сервісу.
- Можлива втрата таблиць при неправильному запуску.

**Умови виникнення:**
- Restart Render Container.
- Deploy нового коду.
- Відсутність або некоректна ініціалізація БД.

---

### 2.2. WSGI сервер (Gunicorn)
**Проблеми:**
- Відсутність gunicorn у requirements.txt → сервер не запускається.
- Некоректний Procfile.

**Умови:**
- Помилка складання (exit 127).
- Неправильна конфігурація середовища.

---

### 2.3. Flask застосунок
**Проблеми:**
- Відсутні шаблони → TemplateNotFound.
- Некоректні імпорти.
- Синтаксичні помилки.

---

### 2.4. Авторизація та сесії
**Проблеми:**
- SECRET_KEY змінюється → користувачі розлогінюються.
- Некоректні дані користувача.

---

### 2.5. Seed-скрипт
**Проблеми:**
- Може дублювати дані.
- Може не виконатися при зміні структури таблиць.

---

### 2.6. Зовнішні сервіси
**Проблеми:**
- Генеровані зображення з Unsplash можуть стати недоступними.
- CDN може повернути 404.

---

## 3. Загальний висновок
Ступінь ризику середній.  
Критичні точки: **SQLite + деплой + відсутність постійного сховища**.

Система працює стабільно в тестових умовах, але потребує переходу на Postgres для production.
